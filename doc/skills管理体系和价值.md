一、背景：为什么需要 skill，skill 是如何解决问题的

1. 为什么需要 skill

之所以会出现 skill 这一层，核心原因在于：模型具备通用能力，但在具体业务场景中，真正决定效果的往往不是模型能否理解任务，而是是否具备完成任务所需的任务能力，即知道该怎么做、按什么边界做、调用什么资源做。

在早期阶段，这类任务能力通常被分散写在：

- Prompt 中
- 操作文档中
- 本地脚本中
- 各类 Tool 封装中
  
这种方式在单个 Agent 项目中尚可运转，但随着 Agent 项目增多，就会暴露出明显问题：

- 同一类任务能力会分散在多个 Agent 项目的 Prompt、文档、脚本和 Tool 封装中，重复建设明显。
- 同一类任务能力由于缺少标准化封装，往往会在多个 Agent 项目中被重复复制、改写和适配，后续逐步分叉。
- 同一类任务能力，会在不同 Agent 中演变为各自独立的实现方式，难以统一管理和复用。
- 由于同一类任务能力分散在多个 Agent 项目中，随着项目数量增加，各个 Agent 项目的维护、升级和排错成本都会持续上升。
  
从管理视角看，这不是单点功能缺失，而是同一类任务能力缺乏统一沉淀、标准化封装和治理机制的问题。

2. skill 是什么

在 Agent 体系中，skill 可以理解为面向某一类任务的标准化能力单元。它通常不是单独的一段 Prompt，也不是单独的一个 Tool 接口，而是把完成某类任务所需的关键要素组织在一起，包括：

- 任务应该如何完成
- 使用时有哪些边界和约束
- 需要参考哪些资料和上下文
- 可以调用哪些脚本、命令或资源
- 输入输出和执行环境有哪些要求
  
从形式上看，skill 把原本分散的知识、经验和执行方式，收敛为一个可复用、可管理、可分发的标准化能力单元。

3. skill 是如何解决问题的

skill 的价值，在于它把原本散落在不同位置的能力信息，升级为标准化能力单元。

具体来说，skill 主要解决了三类问题：

- 把零散的 Prompt、文档、脚本和 Tool 说明收敛为统一的 skill 单元，便于复用和维护。
- 为后续的管理、组合、分发、审核和治理提供一个稳定载体。
  
因此，skill 本质上是 Agent 发展到一定阶段后的自然产物。它的出现，是为了把分散的场景知识和执行经验，从“零散材料”升级为“标准化能力单元”。

二、skills 作为 Agent 数据资产

1. skills 不只是 Prompt，也不只是 Tool 封装

一个完整的 skill，通常不是单一文件，而是一个标准化能力单元，至少会包含以下几类信息：

- 如何完成任务的方法说明
- 使用边界和约束
- 参考资料
- 可直接调用的脚本、命令或资源
  
因此，skills 的本质不是一段 Prompt，也不是一个单独的 Tool，而是“文档+脚本”的标准化封装。

2. 为什么可以把 skills 视为数据资产

从agent建设角度看，skills 具备明确的资产属性：

- 可沉淀：形成后可以长期复用，而不是一次性消耗。
- 可复用：同一份能力可以服务多个 Agent 。
- 可组合：多个 skill 可以形成某个 Agent 的能力集合。
- 可治理：可以做命名、版本、权限、审核和质量控制。
- 可分发：可以安装、同步、共享到不同agent场景。
  
如果模型提供的是通用智能，那么 skills 提供的就是可复用的任务能力。
因此，skills 更适合作为 Agent 的数据资产来建设。

三、skills 在 Agent 生态中的位置

1. skills 在 Agent 技术栈中的生态位

从整体技术栈看，Agent 体系大致可以分为：

1. 模型层：负责通用理解和推理。
2. Agent Runtime 层：负责计划、调度、记忆和调用。
3. skills 数据资产层：负责沉淀领域知识与执行经验。
4. Tool 层：负责具体动作执行。
5. 业务场景层：负责目标定义和结果交付。
  
其中，skills 的作用是承上启下：

- 向上，为模型和 Agent 提供“如何完成任务”的领域化能力。
- 向下，标准化能力单元落实到脚本、命令、文件和接口。
  
因此，skills 所处的不是模型层，也不是执行层，而是在 Agent 技术栈中承担数据资产层的作用。

2. skills 与 Prompt、Tool、Workflow 的区别

形态
主要作用
局限
Prompt
临时引导模型完成一次任务
灵活，但难沉淀、难治理
Tool
提供一个原子动作接口
能执行，但不说明什么时候、怎么用
Workflow
提供一条完整业务流程
闭环强，但较重，复用颗粒度偏大
skill
提供可复用的标准化能力单元
需要标准、治理和分发能力支撑

从生态位看，可以将 skills 理解为：

- 比 Prompt 更稳定
- 比 Tool 更懂任务
- 比 Workflow 更轻，更适合复用
  
这也是为什么 skills 值得被单独建设，而不是被吸收进其它层中。
四.skills 的当前实际进展

1. 智能体应用侧
已经进入真实可用阶段。但都还是以固定skill存放目录，进而使用该目录的skill形式存在
- Codex：已原生支持 skills，并且仍是固定目录式实现，当前主要放在用户级 $HOME/.agents/skills 或项目级 .agents/skills 中，每个 skill 本体仍是带 SKILL.md 的目录。 
- Claude Code：已原生支持 skills，并且仍是固定目录式实现，支持个人目录 ~/.claude/skills/<skill-name>/SKILL.md、项目目录 .claude/skills/<skill-name>/SKILL.md 以及插件目录三种形态。 
- GitHub Copilot：已原生支持 Agent Skills，且仍是固定目录式实现，项目级目录为 .github/skills 或 .claude/skills，个人级目录为 ~/.copilot/skills 或 ~/.claude/skills，skill 内容本质上仍是文件夹内的 SKILL.md 与配套资源。 
- Cursor：已支持 Agent Skills，并且仍是固定目录式实现，以 SKILL.md 文件为入口，官方说明会从 skill directories 自动发现；当前也明确兼容 .cursor/skills/ 以及 Claude、Codex 的 skill 目录。 
- Windsurf：已支持 Skills，且仍是固定目录式实现，工作区目录为 .windsurf/skills/<skill-name>/，全局目录为 ~/.codeium/windsurf/skills/<skill-name>/，入口文件仍是 SKILL.md。
2. 智能体框架侧
目前还没有完全统一。有些框架实现。有些框架并没有实现
- OpenAI API：已原生支持 skills，但实现上更准确地说是“文件包/目录式 skill”而不是依赖某个固定本地扫描目录；skill 本体仍是由 SKILL.md 加配套文件组成的 bundle/folder。 
- Claude API：已原生支持 Agent Skills，skill 本体仍是包含说明、元数据和可选资源的目录化内容；若通过 Agent SDK 本地接入，也仍可从本地目录加载，因此本质上还是目录式 skill，只是不是唯一依赖固定目录扫描。 
- CrewAI：已原生实现 skill system，并且就是固定目录式实现，官方直接将 skill 定义为包含 SKILL.md 的自包含目录。 
- LangChain Deep Agents：已提供原生 skills 机制，并且仍是固定目录式实现，官方明确说明 skills 存放在 skill directories 中，并遵循 Agent Skills specification。 
- LangGraph：核心仍是 graph / state / node / workflow 编排，没有原生固定 skill 目录这一实现要求；能力主要通过节点、边、状态和工具组织，而不是 SKILL.md 目录。 
- AutoGen：核心仍是 agent + tool / workbench / MCP 体系，没有原生固定 skill 目录这一实现方式，主流扩展单位仍是工具和 workbench。 
- Semantic Kernel：当前主要通过 plugin / function calling 扩展能力，不是以固定 skill 目录为原生实现方式，能力封装单位主要是插件和函数。 
- Microsoft Agent Framework：当前以 agent、tool、MCP server 和 workflow 为核心抽象，没有形成原生的固定 skill 目录式实现，重点仍在工具接入与工作流编排。 
3. 当前开源项目进展
- SkillsMP：面向 SKILL.md 生态的聚合型 skill 市场，提供 GitHub 开源 skills 的搜索、分类浏览、一键安装，并支持通过 REST API 做关键字与语义检索。 
- anthropics/skills：Anthropic 的官方 Agent Skills 公开仓库，包含示例 skills、规范 spec 和模板 template，可直接用于 Claude Code、Claude.ai 和 API。 
- skills.pawgrammer.com：社区型 Claude Skills 市场，主打技能发现与分享，用于扩展 Claude 的模块化能力。 
- n-skills：面向多 agent 的精选型 skills 仓库/市场，兼容 Claude Code、Codex 和 openskills，并通过统一仓库做聚合分发。 
- skills.sh：相较 MagicSkills，更偏向“技能分发 + 排行榜 + CLI 安装”，公开文档重点在 npx skills add、leaderboard、匿名安装遥测与基础安全审计，没有像 MagicSkills 那样明确强调共享 skill pool、按 agent 组合 skill 子集、同步到不同 agent 工作流、再通过稳定 tool API 暴露同一组 skills，因此更适合“发现和安装”，不够像完整的多智能体 skill 基础设施。 
- openskills：相较 MagicSkills，更偏向“通用安装器 + AGENTS.md 同步器”，虽然支持从 GitHub、本地路径、私有仓库安装，也支持 sync / read / update，但其默认思路仍是把 skills 装到 ./.claude/skills、./.agent/skills 或全局目录中；README 里甚至专门提供了 symlink 方案来做本地复用，这说明它的核心仍是目录安装与通用加载，而不是像 MagicSkills 那样把“一个共享 skill 池 → 多个命名 Skills 集合 → 面向不同 agent / framework 复用与工具化暴露”作为中心模型。
4. 生态成熟度
当前 skills 生态已经从零散示例阶段进入早期成长期。
- 一方面，SKILL.md + 文件包/目录 已成为公开且被多方采用的主流形态，OpenAI API/Codex 与 Claude API 都已把 skills 作为正式能力机制对外支持。
- 另一方面，生态中已经出现了官方仓库、聚合市场和安装分发基础设施，如 anthropics/skills、SkillsMP、skills.sh，说明 skills 已不只是“能写”，而是初步具备了发现、安装、共享、检索与分发能力。
- 只是从当前实践看，排行榜仍较多依赖安装遥测，安全治理也还主要停留在平台审计与风险提示层面，距离真正成熟的统一治理、稳定版本管理和跨平台标准化复用还有差距。因此，整体上更适合将其判断为：生态雏形已形成，正在快速扩张，但总体仍未进入完全成熟阶段。

五. MagicSkills 当前实际进展
1. 当前已经解决的问题
MagicSkills 当前主要在解决两类现实问题。
第一，当前 skill 在智能体应用侧与智能体框架侧之间，仍缺少统一的组织和接入方式。不同 Agent 往往各自维护一套 skill 结构，导致同类能力难以复用，也不利于形成统一标准。
第二，很多 Agent 在使用 skills 时，仍然采用把 skill 文件夹复制到对应目录的方式来接入能力。这种方式短期可用，但长期会带来重复建设、内容分叉、维护困难和跨运行时适配成本上升等问题。
围绕上述问题，MagicSkills 当前已经实现了以下几方面的实际作用：
-  让同一份能力可以被多个 Agent 复用，而不是在不同项目中重复复制 
-  让不同 Agent 只暴露各自所需的 skill 子集，减少上下文冗余和管理复杂度 
-  覆盖并统一智能体应用与智能体框架两侧的 skills 接入方式，同时兼容 AGENTS.md 型接入和 tool / function-call 型接入，降低跨运行时适配成本 
-  为后续的 skill 共享、上传、审核、扫描和治理提供统一入口 
因此，从当前阶段看，MagicSkills 的现实意义主要体现在三点：
-  降低重复建设成本 
-  提高跨 Agent 复用效率 
-  为后续平台化建设提供标准化能力单元

（1）已经形成基础设施雏形
MagicSkills 当前可以概括为：一个面向多 Agent 项目、以 SKILL.md 为组织核心、本地优先的 skills 基础设施。仓库 README 也明确使用了“local-first skill infrastructure for multi-agent projects”这一定位。
在这一定位下，它已经开始解决 skill 生命周期中的一些核心基础问题，包括：

-  skill 如何进入共享 skill 池 
-  skill 如何按不同 Agent 进行组合

（3）已经具备可运行的项目基础
从当前公开仓库来看，MagicSkills 已经具备较明确的项目基础，而不是停留在概念阶段：

后续投入可以重点放在三个方向：

-  远程 registry / catalog

-  版本和依赖管理 
-  权限、审核、审计、状态管理 
二是高价值 skill 建设。

 随着 skills 越来越多，平台能不能稳定发展，很大程度上取决于治理能力。
 这一块重点包括：
-  风险扫描 
-  权限控制和沙箱机制 
-  效果评测和基准集 
-  调用日志和使用反馈 
-  基于数据的升级和淘汰机制

AI易懂精简版
一、我们遇到了什么问题
我在做 AI 智能体（Agent）项目时发现一个反复出现的痛点：
同样的能力，每个项目都要重新写一遍。
举个例子，"代码审查"这件事，A 项目写了一套提示词，B 项目又写了一套脚本，C 项目还有一套文档。它们做的事情几乎一样，但各自独立维护，互不复用。
随着项目越来越多，问题越来越严重：
暂时无法在飞书文档外展示此内容
根本原因只有一个：缺少一个统一的能力管理层。

---
二、什么是 Skill
为了解决上述问题，我引入了 Skill（技能） 这个概念。
一句话解释
Skill 就是把完成某类任务所需的全部知识打包成一个标准化单元，让多个智能体都能直接使用。
一个 Skill 里通常包含什么
- 📋 做法说明 — 这个任务该怎么一步步完成
- 🚧 使用边界 — 什么场景适用、什么情况不能用
- 📚 参考资料 — 需要查阅的文档和背景知识
- ⚙️ 可执行资源 — 配套的脚本、命令或接口
- 📐 输入输出规范 — 需要什么输入、产出什么结果
打个比方
如果把智能体比作一个员工，那么：
- 模型 = 员工的智商和理解力
- Skill = 公司的标准操作手册（SOP）
- Tool = 员工手边的具体工具
员工再聪明，没有操作手册，也不知道该怎么干活。Skill 就是那本操作手册。

八、为什么值得继续投入
第一，它是公共能力底座
MagicSkills 不是服务某一个项目的小工具。它是在为多个智能体共用能力打基础。一次建设，多处受益。
第二，越早建设，后续成本越低
如果现在不做统一的 Skill 层，等项目数量上来后：
- 每个项目各自积累一套能力 → 重复建设
- 相同能力多处实现 → 版本分叉
- 后期想整合 → 成本极高
提前投入，本质上是在降低未来的系统性成本。
第三，有清晰的平台化空间
MagicSkills 未来可以发展为完整的 Skill 资产平台：

十、总结
三个核心结论：
1. Skill 值得作为独立方向投入 — 它承载的是智能体的可复用能力，是数据资产层
2. MagicSkills 已完成基础设施验证 — 统一管理、统一分发、跨智能体复用，工程上已经走通
3. 下一步是升级为资产平台 — 不是补零散功能，而是围绕平台、内容和治理做专项建设
两个关键决策点：
- 是否将 Skill 定位为多智能体体系的共性能力底座，而不是单项目配套工具？
- 是否给予专项资源，推进 Skill 资产管理、内容供给、质量治理 三条主线？
模型决定了智能体"能不能理解"，而 Skill 决定了智能体"会不会做事"。MagicSkills 要做的，就是让每一个智能体都不用从零学起。
